## The Development Lifecycle

Professional teams follow a lifecycle. So will we:

```
1. Strategy → 2. Design → 3. Implementation → 4. Security → 5. Release
      ↑                                                          |
      └──────────────── Learn & Iterate ─────────────────────────┘
```

### Phase 1: Strategy
> *"What are we building and why?"*

**Skill**: [Product Strategist](./skills/product-strategist/SKILL.md)

Before writing code, we'll define:
- What problem are we solving?
- How will we know if it worked?
- What could go wrong?

**Standards**: [PRD Template](./standards/prd-template.md)

---

### Phase 2: Design
> *"How should this look and feel?"*

**Skill**: [Design Systems Lead](./skills/design-systems-lead/SKILL.md)

We'll think through:
- Visual hierarchy (what should users see first?)
- Accessibility (can everyone use this?)
- Consistency (does it feel like part of the same product?)

**Standards**: [Accessibility Checklist](./standards/accessibility-check.md)

---

### Phase 3: Implementation
> *"Let's build it right"*

**Frontend**: [React System Builder](./skills/react-system-builder/SKILL.md)
**Backend**: [Supabase Architect](./skills/supabase-architect/SKILL.md)

We'll focus on:
- Clean, maintainable code
- Performance from the start
- Security baked in, not bolted on

**Standards**: [React Patterns](./standards/react-patterns.md)

---

### Phase 4: Security
> *"Is this safe to ship?"*

**Skill**: [Security Gatekeeper](./skills/security-gatekeeper/SKILL.md)

We'll check:
- Authentication & authorization
- Data protection
- Common vulnerabilities

**Gate**: Code doesn't merge until security review passes.

---

### Phase 5: Release
> *"Ship it with confidence"*

**Skill**: [Release Engineer](./skills/release-engineer/SKILL.md)

We'll ensure:
- Automated tests catch regressions
- Performance meets targets
- Rollback plan exists

**Performance Gates**:
| Metric | Target |
|--------|--------|
| Lighthouse Performance | ≥90 |
| Accessibility | 100 |
| LCP | <2.5s |

## Agent Behavior

When helping you, I will:

1. **Coach before coding** — I'll ask questions to understand what you're really trying to do
2. **Explain the "why"** — You'll understand the reasoning, not just follow instructions
3. **Challenge when needed** — If something seems unclear, I'll push back (respectfully)
4. **Reference the masters** — "Here's how Google approaches this..." / "Apple would say..."
5. **Meet you where you are** — Beginner or expert, I'll adjust my explanations
